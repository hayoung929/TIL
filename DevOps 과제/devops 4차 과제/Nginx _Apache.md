사용함# Nginx , Apache
## 웹 서버
<details>
<summary>웹 서버의 의미</summary>

- 개념
  - 웹 서버는 HTTP 프로토콜을 이용해 클라이언트로부터 요청을 받아 HTML,CSS,이미지 같은 정적 웹 페이지 파일을 찾아서 보내주는 서버
  - 물리적인 컴퓨터 자체를 가리킬 때도 있고, **Apache,Nginx** 같은 웹 서버 프로그램을 지칭 할 때도 있다.
- 특징
  - **HTTP 기반** : 주로  HTTP/HTTPS 프로토콜로 통신하며, 요청과 응답 메시지 구조를 따른다
  - **정적 콘텐츠** 중심 : HTML, CSS, JS, 이미지 등 정적 웹 페이지 파일 제공이 기본 역할이다.
  - **클라이언트-서버구조** : 클라이언트의 요청을 받아 처리하고 응답을 보내는 구조
  - **무상태성(stateless)** : HTTP 특성상 각 요청은 독립적으로 처리되고, 연결은 응답 후 끊어지는 형태가 기본이다.
  - 확장 기능 : 로그 기록, 보안, 압축, 캐시, 가상 호스팅 등 성능, 안정성을 위한 기능 제공
  - **고부하 처리** : 수십~수만 건의 동시 요청을 효율적으로 처리하도록 설계 되며, 대용량 트래픽 환경에서 많이 사용함

### 보충내용
1. **정적 콘텐츠**
- 개념 : 서버에 미리 저장되어 있고, 누가 봐도 내용이 똑같이 나오는 파일들
- ex : HTML, CSS, JS, 이미지 파일 같은 것들?
- 특징
  - 요청이 오면 서버는 해당 파일을 그대로 보내면 됨
  - 같은 주소로 접속하면, 한국에서 보든 같은 화면이 보임
2. 웹 서버의 확장 기능
- **로그 기록**
  - 언제, 어떤 IP가, 어떤 페이지를 요청했는지 파일에 기록
  - 트래픽 분석, 에러 분석에 사용
- **보안**
  - HTTPS 지원해서 중간에서 엿보지 못하게 함
  - 특정 디렉터리에 비밀번호를 걸어 접근 제한 가능
- **압축 전송**
  - gzip, brotli 같은 걸로 HTML/CSS/JS를 압축해서 전송, 속도 향상
- **캐시 설정**
  - 정적 파일에 Cache-Control, Expires 헤더를 붙여서, 브라우저나 중간 서버가 파일을 저장해 두고 재사용하게 함.
- **가상 호스팅(Virtual Hosting)**
  - 한 IP/한 포트에서 example.com, test.com 처럼 여러 도메인 사이트를 동시에 서비스
3. 고부하 처리 설계
- **동시 처리 구조**
  - 여러 요청을 한 번에 처리할 수 있도록 프로세스/쓰레드/이벤트 루프 같은 구조 사용
  - **프로세스(Process)**
    - 운영체제가 실행 중인 프로그램 하나를 부를 때 쓰는 단위
    - 자기만의 메모리 공간을 가지고 있어서, 다른 프로세스랑 메모리를 직접 공유하지 않음
  - **쓰레드(Thread)**
    - CPU에 작업 요청을 하는 실행단위를 말한다
    - 하나의 프로세스 안에 쓰레드를 여러 개 두면, 동시에 여러 일을 하는 것처럼 처리할 수 있음
    - 같은 프로세스 안의 쓰레드들은 메모리를 같이 쓰기 때문에, 데이터 공유가 빠르지만 동기화가 필요함
  - **이벤트 루프**
    - 이벤트/요청을 줄 세워 놓고, 하나씩 꺼내서 처리하는 반복 구조
    - Nginx는 이벤트 루프 구현에 epoll(Linux), kqueue(macOS/BSD)를 사용
      - OS 레벨에서 소켓 목록을 등록해두면, OS가 변화가 생긴 소켓만 감지하여 Nginx에 통보
      - 이벤트 루프는 통보받은 소켓만 꺼내서 처리
    - 덕분에 쓰레드를 많이 만들지 않고도, 여러 요청을 동시에 처리하는 것처럼 보이게 만들 수 있음
- 캐시와 정적 파일 최적화
  - 정적 콘텐츠는 디스크에서만 읽지 않고, 메모리 캐시나 OS캐시를 적극 활용
- 서버 성능 확장
  - 스케일 업 : 한 서버에 CPU, RAM 등을 더 넣어서 "한 대를 더 세게" 만드는 방식
  - 스케일 아웃 : 웹 서버를 여러 대 두고, 앞에 로드 밸런서를 두어 트래픽을 나눠 받는 방식

### 웹서버의 역할
- 클라이언트로부터 HTTP 요청을 받고 HTML,CSS,이미지 같은 웹 리소스를 응답으로 돌려준다.
- 요청이 정적 콘텐츠면 직접 파일을 찾아서 보내고, 동적 처리 필요하면 WAS 같은 애플리 케이션 서버로 요청을 넘긴다
<details>
<summary>WAS</summary>

- **Web Application Server**
- 위치 : 웹 서버와 DB 서버 사이에서 동작하는 미들웨어 역할
- 역할
- 사용자의 요청에 맞게 비즈니스 로직을 실행
- 필요하면 데이터베이스에 쿼리를 날려서 데이터를 읽고 저장
    - 결과를 가지고 HTML,JSON 같은 동적 콘텐츠를 만들어 웹 서버 통해 사용자에게 응답
</details>

- 접속 로그를 남기고, SSL/HTTPS, 인증 접근 제어 등 보안 기능 제공
### 웹 서버의 작동 과정
1. **주소 입력 / 링크 클릭**
- 사용자가 브라우저에 URL을 입력하거나 링크를 클릭하면, 브라우저가 해당 서버(IP,포트)로 HTTP 요청을 보낸다.
2. **요청 수신 및 해석**
웹 서버는 지정된 포트에서 요청을 받고 메서드,경로,헤더 등을 분석한다
3. **리소스 찾기 또는 위임**
- 요청 경로에 맞는 파일을 문서 루트에서 찾음
- 동적 페이지면 WAS에 요청을 전달한다
4. **HTTP 응답 생성**
- 상대 코드, 헤더, 응답 본문을 만들어 클라이언트로 보냄
5. **연결 정리**
- HTTP/1.1에서는 keep-alive를 통해 일정 시간 같은 연결을 재사용할 수 있고, HTTP/2 이상에서는 하나의 연결 위에서 여러 요청을 다룸
### 포트 번호와 웹 서버
- IP는 "어떤 컴퓨터인가"를 나타내고, 포트는 "그 컴퓨터 안의 어떤 프로그램인가"를 나타내는 논리적인 번호
- 같은 IP 위에서 웹, 메일, SSH 같은 여러 서비스가 동시에 돌아가기 때문에, 패킷을 어느 프로그램에게 넘길지 구분하려고 포트를 사용함
</details>

## Apache
<details>
<summary>Apache 개념</summary>

- Apache 뜻
  - **Apache HTTP Server**, 보통 아파치라고 부르는 **오픈소스 웹 서버 소프트웨어** 
  - Linux/Unix/Windows/MaxOS 등 **대부분 OS에서 동작**
- Apache 장점
  - **오픈소스, 무료**: 라이선스 비용 없이 자유롭게 사용 가능.
  - **모듈**식 구조 : 필요에 따라 모듈을 켜고 끄면서 기능을 유연하게 확장 가능
  - **높은 호환성** : 다양한 OS,언어(PHP,Perl,Python 등)와 잘 연동됨
  - **가상 호스팅** 지원 : 한 서버에서 여러 웹사이트 운영 가능
  - 커뮤니티/자료 풍부 : 오래되고 사용자가 많아서 설정 예제, 문서, 튜토리얼이 매우 많음
- Apache 단점
  - **높은 동시접속에서 비효율적**일 수 있음: **프로세스/쓰레드 기반 MPM** 때문에 아주 많은 동시 접속에서 Nginx 같은 이벤트 기반 서버보다 메모리 사용이 커질 수 있음
  - 설정이 복잡해질 수 있음 
    - 사이트/서비스가 많아질 때
    - 성능 튜닝을 직접 하려고 할때
    - 보안/접근 제어를 세밀하게 걸때
    - 모듈을 많이 쓸 때
    - 설정 파일이 분산될 때
  - **Nignx의 등장으로 점유율 일부 감소 추세**
- Apache 모듈
  - 모듈 개념
    - 모듈은 **특정 기능을 담당하는 플러그인 같은 코드 조각**
    - LoadModul 지시어로 필요한 모듈만 로드해서 메모리 사용과 기능 조절
<details>
<summary>대표적인 모듈</summary>

- **MPM 모듈(Mult-Processing-Module)**
  - **prefork**
    - 개념 : 멀티 프로세스, 쓰레드 없이 프로세스만으로 요청 처리하는 MPM
    - 특징 : 안정적이고 레거시 호환이 좋지만, 동시 접속 많을수록 메모리 사용이 크다.
  - **worker**
    - 개념 : 여러 프로세스 안에서 여러 쓰레드가 요청을 처리하는 MPM
    - 특징 : 메모리를 덜 쓰면서 많은 동시 접속을 처리할 수 있지만, 쓰레드 안전이 요구됨
  - **event**
    - 개념 : worker 구조에 이벤트 기반을 더해서 keep-alive 연결을 효율적으로 처리하는 MPM
    - 특징 : 유휴 keep-alive 연결에 쓰레드를 잡아두지 않아 고동시,고트래픽 환경에 유리하다.
- **요청/응답 관련 모듈**
  - **mod_alias**
    - 개념 : 단순한 URL 리다이렉트와 경로 매핑을 담당하는 모듈
    - 특징 : 문법이 단순하고 가벼워, 기본적인 redirect/alias 설정에 적합하다.
  - **mod_rewrite**
    - 개념 : 정규식을 이용해 URL을 제작성하거나 복잡한 리다이렉트를 처리하는 모듈
    - 특징 : 매우 강력하고 유연하지만, 규칙이 복잡해지면 설정 난이도가 높다.
  - **mod_headers**
    - 개념 : 요청/응답의 HTTP 헤더를 추가,수정,삭제하는 모듈
    - 특징 : 보안 헤더, CORS, 캐시 정책 등 헤더 기반 설정을 세밀하게 제어할 수 있다.
- **보안/인증 관련 모듈**
  - **mod_ssl**
    - 개념 : Apache에 SSL/TLS 기능을 추가해 HTTPS를 제공하는 모듈
    - 특징 : 인증서 설정과 암호 스위트 제어를 통해 443 포트 HTTPS 서비스를 구성한다.
  - **mod_auth_basic**
    - 개념 : HTTP Basic 방식으로 ID/비밀번호 인증을 제공하는 모듈
    -  특징 : 설정이 간단하지만, 반드시 HTTPS와 함께 써야 안전하다.
  - **mod_auth_digest**
    -** 개념 : HTTP Digest 방식 인증을 제공해 패스워드를 해시 기반으로 검증하는 모듈
    - 특징 : Basic보다 안전하지만, 요즘엔 애플리케이션 레벨 인증에 밀려 사용 빈도가 낮다.
- **프록시/캐시 관련 모듈**
  - **mod_proxy/mod_proxy_http/mod_proxy_balancer**
    - 개념 : Apache를 프록시/리버스 프록시/로드밸런서로 만들어 주는 모듈들
    - 특징 : 백엔드 서버 여러 대로 요청을 분산하거나, 특정 경로를 다른 서버로 넘기는 게이트웨이 구성을 가능하게 한다.
  - **mod_cache / mod_cache_disk**
    - 개념 : 응답을 캐시로 저장해 두었다가 재사용하는 캐시 기능 모듈
    - 특징 : 자주 조회되는 응답을 디스크 등에 저장해, 백엔드 부하를 줄이고 응답 속도를 높여준다.
- **SSL**
  - **Secure Sockets Layer**
  - 인터넷 통신을 암호화해서 중간에서 패킷을 훔쳐봐도 내용을 알아볼 수 없게 만드는 보안 프로토콜
  - TLS는 SSL의 업그레이드 버전
  - HTTP + TLS = HTTPS 
</details>

## Nginx
<details>
<summary>Nginx</summary>

- **Nginx**는 고성능/경량 오픈소스 웹 서버이자 **리버스 프록시 서버**이다.
- ****비동기 이벤트 기반 아키텍처**를 사용해, 소수의 프로세스로 매우 많은 접속을 처리하도록 설계됨
- 장점
  - **높은 동시 접속 처리** : 이벤트 루프 기반이라 트래픽이 몰려도 메모리 사용량이 비교적 적고, 요청을 효율적으로 처리
  - **정적 콘텐츠 성능 우수** : 이미지,CSS,JS 같은 정적 파일을 매우 빠르게 전달
  - **리버스 프록시/로드밸런서에 특화** : 여러 WAS로 요청을 분산하고 SSL 종료, 캐싱, 압축 등을 앞단에서 처리하는게 용이
  - **가벼운 자원** 사용: 프로세스/쓰레드 수가 적어, 같은 동시 접속에서 Apache보다 메모리를 적게 쓴다
- 단점
  - **.htaccess 미지원** : 디렉터리별 분산 설정이 없고, 모든 설정을 중앙 conf에서 관리해야 해서 공유 호스팅 환경에서는 불편할 수 있음
  - Apache보다 덜 유연한 동적 모듈 구조: 전통적으로 모듈을 정적으로 빌드해야 했고, 동적으로 붙였다 떼는 부분은 Apache만큼 자유롭지 않음
  - 설정 문법이 처음엔 낯설 수 있음: **블록 구조**(http, server, location)에 익숙해지는 시간이 필요
</details>

<details>
<summary>Nginx vs Apache</summary>

1. **아키텍처** 차이
- **Apache**
  - **프로세스/쓰레드 기반: MPM**(prefork, worker, event)으로 요청을 처리하는 방식 결정.
  - 연결 수가 늘어나면 프로세스/쓰레드 수가 늘고, 메모리 컨텍스트 스위칭 비용이 증가할 수 있음.
- **Nginx**
  - 비동기 이벤트 기반: **소수의 워커 프로세스**가 **이벤트 루프**를 돌면서 많은 연결을 동시에 처리
  - I/O 대기 중인 연결에 쓰레드를 묶어두지 않아서, **고동시 접속 환경**에서 특히 강함
2. **정적/동적 콘텐츠 처리**
- **정적 콘텐츠**
  - **Apache**
    - 정적 파일 처리 가능하지만, 구조상 Nginx보다 메모리 효율이 떨어질 수 있음
  - **Nginx**
    - 정적 파일 서빙이 매우 빠르고, CPU·메모리 사용량도 효율적. CDN 프론트처럼 쓰기 좋음
- **동적 콘텐츠**
  - **Apache**
    - mod_php, CGI, 또는 리버스 프록시로 WAS와 직접 연동 등 다양한 방식 지원, 레거시 PHP 환경에 강함
  - **Nginx**
    - 동적 처리는 직접 하지 않고, FastCGI(php-fpm), uwsgi, 혹은 뒤쪽 WAS(Spring, Node 등)로 넘기는 역할이 중심
3. **설정과 확장성**
- **Apache**
  - 모듈 생태계가 매우 풍부하고, .htaccess 로 디렉터리 단위 설정 override 가능 
  - 대신 설정이 여러 파일로 분리되고, **복잡한 모듈 조합에서는 전체 구조 파악이 어려워질 수 있음**
- **Nginx**
  - 중앙 설정 파일 + server/location 블록 기반 구조, 상대적으로 단순한 계층 구조
  - 디렉터리 분산 설정은 없지만, 앞단 게이트웨이/프록시 역할로 설정을 한 곳에서 관리하기 좋음
4. 사용하는 경우
- **Apache good**
  - 레거시 PHP/CGI 프로젝트, 공유호스팅, .htaccess가 많이 쓰이는 환경
- **Nginx good**
  - 대규모 트래픽, 정적 파일 많고 API 서버가 뒤에 여러 대 있는 구조, 리버스 프록시·로드밸런서 필요
- **같이**
  - 클라이언트 -> Nginx -> Apache/WAS -> DB
  - Nginx가 앞단에서 SSL 종료, 정적 파일, 캐시, 로드밸런싱을 처리하고, 동적 요청만 Apache/Tomcat/Spring 같은 WAS로 전달하는 구조가 흔함
</details>

<summary>Nginx의 설정 파일 구조</summary>

- **/etc/nginx/nginx.conf**
  - Nginx 전역 설정 파일, 전체 서버 동작 방식 정의
- **/etc/nginx/conf.d/**
  - 추가 설정 파일 디렉터리, 이 안의 *.conf 는 자동으로 불러옴
  - 공통 설정(업스트림, 공용 헤더, 공용 프록시 설정 등)을 나누어 두기 좋음
- **/etc/nginx/sites-available/**
  - 개별 사이트(도메인, 서비스)용 server 블록 설정 파일을 저장하는 곳
- **/etc/nginx/sites-enabled/**
  - 실제로 활성화된 사이트 설정 파일에 대한 심볼릭 링크가 위치
  - 보통 sites-available 에서 ln -s(또는 nxensite 스크립트)로 연결해서 사용
- **/var/www/ (/var/www/html)**
  - 웹 사이트 실제 파일(HTML, CSS, JS, 이미지 등)이 위치하는 기본 웹 루트 디렉터리
- **로그 디렉터리** (/var/log/nginx)
  - access.log, error.log 등 Nginx 접근 로그와 에러 로그가 저장되는 위치
</details>
